# 02. 代码规范（Code Standards）

## 1. 基本原则
- 小步提交：每次改动只解决一个明确问题。
- 最小侵入：优先局部修复，避免连带重构。
- 可追溯：关键决策记录在 PR/任务说明或文档。

## 2. 命名与文件组织
- 组件文件：`PascalCase.jsx`（例：`LicenseReportWindow.jsx`）。
- 状态/服务：`camelCase.js`（例：`licenseStore.js`, `graphLoader.js`）。
- 目录按领域而非技术分层，新增模块优先落在现有域目录。

## 3. 事件驱动约束
- 事件名使用动词过去式或意图式：`graphLoaded`、`downloadGraphRequested`。
- 每个新事件必须写明：触发条件、载荷结构、订阅者。
- 禁止“隐式全局副作用”：事件处理函数应显式更新目标 Store。

## 4. 状态与渲染
- Store 负责状态，不直接拼装 UI 文本。
- 组件负责展示，不直接耦合底层图数据结构细节。
- 高频交互路径（hover、drag、zoom）避免引入同步重计算。

## 5. 错误处理
- 服务层必须兜底：网络异常、空图数据、解析失败。
- 用户可见功能失败时，至少提供 console 可定位信息。
- 非阻断错误优先降级处理，不直接让主场景崩溃。

## 6. 注释与文档
- 仅在复杂逻辑前写短注释，说明“为什么”。
- 新增公共接口（store/service/runtime）需同步更新文档。
- 文档优先更新到 `docs/VIBECODING/` 或 `src/docs/`。

## 7. 提交前检查
- 本地通过 `npm run check:vibe`。
- 手动验证最短主链路：打开图谱、搜索、选择节点、打开一个窗口。
- 记录已知限制（如果有）。
